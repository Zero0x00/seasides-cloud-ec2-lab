Vulnerability

SSRF (Server-Side Request Forgery) is a dangerous web vulnerability caused by poor programming practices. SSRF allows attackers to send requests from the server to other resources. Attackers typically use SSRF attacks to target internal systems that are behind firewalls and are not accessible from the external network.

How SSRF Works:

  1. The attacker identifies an application where a URL can be imported or where data is read from a URL.
  2. The attacker attempts to modify the URL by providing a different URL or manipulating the existing one.
  3. Once the manipulated URL is sent to the server, the server-side code tries to read data from the manipulated URL, allowing the attacker to potentially access services.

How to Prevent SSRF:

  1. Never trust user input.
  2. Implement whitelisting of domains and IP addresses.
  3. Enable authentication on all services.
  4. Sanitize and validate inputs.

SSRF Payloads:

SSRF with Blacklisted-Based Input Filters

  1. Visit a product page, click "Check stock," and intercept the request in Burp Suite, then send it to Burp Repeater.
  2. Change the URL in the "stockApi" parameter to "http://127.0.0.1/" and observe that the request is blocked.
  3. Bypass the block by changing the URL to "http://127.1/".
  4. Change the URL to "http://127.1/admin" and observe that the URL is blocked again.
  5. Obfuscate the "a" by double-URL encoding it to "%2561" to access the admin interface and delete the target user.

SSRF with Filter Bypass via Open Redirection Vulnerability

  1. Visit a product page, click "Check stock," and intercept the request in Burp Suite, then send it to Burp Repeater.
  2. Try tampering with the "stockApi" parameter and observe that it isn't possible to make the server issue the request directly to a different host.
  3. Click "next product" and observe that the "path" parameter is placed into the Location header of a redirection response, resulting in an open redirection.
  4. Create a URL that exploits the open redirection vulnerability and redirects to the admin interface, then provide this URL as the "stockApi" parameter on the stock checker: "/product/nextProduct?path=http://192.168.0.12:8080/admin".
  5. Observe that the stock checker follows the redirection and shows you the admin page.
  6. Amend the path to delete the target user: "/product/nextProduct?path=http://192.168.0.12:8080/admin/delete?username=carlos".
